# Mobile Integration Guide

This document is a complete, self-contained reference for integrating a mobile client with the Sabil Life backend. It is written so a mobile agent can implement against it without reading the Django source.

If anything here disagrees with the live OpenAPI schema, the OpenAPI schema wins — fetch it from `/api/schema/` (see [Live schema](#live-schema)).

---

## 1. Conventions

### 1.1 Base URL and versioning

| Environment | Base URL                       |
| ----------- | ------------------------------ |
| Local dev   | `http://localhost:8000`        |
| Staging     | _TBD_                          |
| Production  | _TBD_                          |

All API routes are versioned under `/api/v1/`. There is currently only `v1`. The versioning style is **URL path versioning** — versions are part of the path, not a header.

### 1.2 Content type

- Request: `application/json`
- Response: `application/json`
- Files (e.g. listing images): the backend does **not** accept multipart uploads at the moment. `image_urls` is a JSON array of pre-uploaded URLs the client is responsible for producing (e.g. via S3 presigned upload from the mobile app). Talk to backend before adding a server-side upload endpoint.

### 1.3 Authentication header

```
Authorization: Bearer <token>
```

Tokens are issued by **django-rest-knox** (not JWT, single opaque token, server-side hashed). Properties:

- Length: 64 chars
- TTL: 10 hours from issue
- No refresh token — when the token expires the user logs in again
- One user can have multiple active tokens (different devices)
- `/auth/logout/` invalidates the current token; `/auth/logoutall/` invalidates all of the user's tokens

### 1.4 Pagination

All list endpoints use **page-number pagination** with a default page size of **20**:

```json
{
  "count": 137,
  "next": "https://api.example.com/api/v1/listings/?page=3",
  "previous": "https://api.example.com/api/v1/listings/?page=1",
  "results": [ ... ]
}
```

Query params:
- `?page=N` — fetch a specific page
- `?page_size=N` — _not currently supported_; the page size is fixed at 20 server-side. If mobile needs a different size, ask the backend team.

### 1.5 Error shape

Standard DRF errors:

| Status | Body shape                                          | Meaning                            |
| ------ | --------------------------------------------------- | ---------------------------------- |
| 400    | `{"<field>": ["msg", ...]}` or `{"detail": "msg"}`  | Validation / bad request           |
| 401    | `{"detail": "Authentication credentials were not provided."}` | Missing or expired token  |
| 403    | `{"detail": "You do not have permission..."}`       | Role/permission denied             |
| 404    | `{"detail": "Not found."}`                          | Resource missing or hidden         |
| 409    | `{"detail": "..."}`                                 | State-machine conflict, duplicate  |
| 429    | `{"detail": "Request was throttled. Expected available in N seconds."}` | Rate limit |
| 500    | _opaque_                                            | Server error (logged to Sentry)    |

Treat **404 as "either does not exist or you don't own it"** — the backend deliberately returns 404 instead of 403 to avoid leaking the existence of other users' resources. Don't show "permission denied" UI for 404s.

### 1.6 Rate limits

| Scope         | Limit         | Applies to                                   |
| ------------- | ------------- | -------------------------------------------- |
| `anon`        | 100 / day     | Unauthenticated requests (registration, public listings) |
| `user`        | 1000 / day    | All authenticated endpoints                  |
| `user_login`  | 5 / minute    | `POST /auth/login/`                          |

On `429`, back off using the `Retry-After` header if present, otherwise exponential.

### 1.7 IDs

- **User id**: integer (auto-pk).
- **Listing, Inquiry, MasterclassSubscription, Review id**: UUID v4 string. The string form is used everywhere — never assume hyphenated/non-hyphenated; pass it back verbatim.
- **ServiceSuggestion id**: integer.
- **Category keys, status values**: uppercase strings (e.g. `"TUTORING"`, `"PENDING"`). Treat them as enums; the server may send back lowercase tolerated values for filters but **always returns uppercase** in payloads.

### 1.8 Timestamps

All timestamps are ISO 8601 UTC, with a trailing `Z` or `+00:00`. Parse defensively.

### 1.9 Distances and money

- `distance_km`: float, rounded to 2 decimals; `null` when not annotated or when the listing has no coordinates.
- `price_from_qar`: integer (whole QAR), never decimal. `0` means "free / contact provider".
- `rating`: decimal string with 1 decimal, e.g. `"4.7"`. Don't parse as int.

## 3. Personas and permissions

The system has four user roles. Mobile flows differ per role.

| Role          | Who                                  | Verified by                          |
| ------------- | ------------------------------------ | ------------------------------------ |
| `FAMILY`      | Parents browsing / booking           | Auto-verified on registration         |
| `TUTOR`       | One-on-one tutor (creates `TUTORING` listings) | Admin must flip `is_verified` |
| `MASTERCLASS` | Class/workshop provider (creates `MASTERCLASSES` listings) | Admin must flip `is_verified` |
| `ADMIN`       | Internal staff                       | Created via `createsuperuser`        |

**Rules to encode in the mobile app:**

- Mobile **cannot register an admin** — `role=ADMIN` is rejected with 400.
- A newly registered `TUTOR`/`MASTERCLASS` user has `is_verified=false`. They can use the app but listings they create start in `DRAFT` and are not publicly visible until verified by an admin. Show an "awaiting verification" UI state for these users until `me/` reports `is_verified=true`.
- A `TUTOR` can only create listings with `category="TUTORING"`. A `MASTERCLASS` can only create listings with `category="MASTERCLASSES"`. The server enforces this — your form should grey out other categories.
- `FAMILY` and `TUTOR`/`MASTERCLASS` flows are completely separate:
  - Family endpoints live under `/api/v1/<resource>/`
  - Provider endpoints live under `/api/v1/provider/<resource>/`
  - Hitting the wrong namespace with the wrong role returns `403`.

## 4. Endpoint catalog

Routes are grouped by feature area, mirroring the backend apps.

### 4.1 Auth & user profile (`apps/users`)

| Method | Path                       | Auth        | Purpose                                  |
| ------ | -------------------------- | ----------- | ---------------------------------------- |
| POST   | `/api/v1/auth/register/`   | none        | Self-service signup, returns token       |
| POST   | `/api/v1/auth/login/`      | none        | Email+password → token                   |
| POST   | `/api/v1/auth/logout/`     | bearer      | Invalidate current token                 |
| POST   | `/api/v1/auth/logoutall/`  | bearer      | Invalidate all tokens for this user      |
| GET    | `/api/v1/auth/me/`         | bearer      | Current user profile                     |
| PATCH  | `/api/v1/auth/me/`         | bearer      | Update profile fields (incl. password)   |
| PUT    | `/api/v1/auth/me/`         | bearer      | Full profile replacement                 |

#### POST `/api/v1/auth/register/`

Request:
```json
{
  "email": "parent@example.com",
  "password": "at-least-8-chars",
  "full_name": "Aisha Al-Thani",
  "role": "FAMILY",
  "phone": "+97455123456"
}
```

- `role` is optional, default `FAMILY`. Accepts `FAMILY` / `TUTOR` / `MASTERCLASS`. Rejects `ADMIN`.
- `full_name` and `phone` are optional and default to empty strings.

Response **201**:
```json
{
  "user": {
    "id": 42,
    "email": "parent@example.com",
    "full_name": "Aisha Al-Thani",
    "role": "FAMILY",
    "is_verified": true,
    "phone": "+97455123456",
    "home_lat": null,
    "home_lng": null,
    "first_name": "",
    "last_name": ""
  },
  "token": "0d3f...64chars",
  "expiry": "2026-06-20T09:00:00Z"
}
```

Errors:
- 400 — missing/invalid email, password validation failure (`{"password": ["This password is too short..."]}`), `role=ADMIN`
- 429 — anonymous rate limit

#### POST `/api/v1/auth/login/`

Request:
```json
{
  "email": "parent@example.com",
  "password": "..."
}
```

Response **200**:
```json
{
  "expiry": "2026-06-20T09:00:00Z",
  "token": "0d3f...",
  "user": { /* same shape as register */ }
}
```

Errors:
- 400 — `{"non_field_errors": ["Unable to log in with provided credentials."]}` for bad creds, deactivated user, etc.
- 429 — login rate limit (5/min)

#### GET `/api/v1/auth/me/`

Response **200**: same `user` shape as above.

#### PATCH `/api/v1/auth/me/`

Editable fields: `full_name`, `phone`, `home_lat`, `home_lng`, `first_name`, `last_name`, `password`.
Read-only: `id`, `email`, `role`, `is_verified`.

To change password, include `"password": "newvalue"` in the PATCH body. The server re-hashes and persists immediately. **Existing tokens are not invalidated** on password change — issue a `logoutall` afterward if the UX expects it.

#### POST `/api/v1/auth/logout/` and `/logoutall/`

No body. Response **204**.

### 4.2 Public catalog (`apps/catalog`)

These endpoints are **unauthenticated** — used by both anonymous browsing and logged-in users. Only listings with `status="ACTIVE"` are returned.

| Method | Path                                | Purpose                                   |
| ------ | ----------------------------------- | ----------------------------------------- |
| GET    | `/api/v1/listings/`                 | Paginated list, with filters / sort       |
| GET    | `/api/v1/listings/{id}/`            | Listing detail + 10 most recent reviews   |
| GET    | `/api/v1/categories/`               | All categories with active counts         |

#### GET `/api/v1/listings/`

Query params:

| Param              | Type    | Notes                                                   |
| ------------------ | ------- | ------------------------------------------------------- |
| `category`         | string  | Case-insensitive: `TUTORING`, `tutoring`, etc.          |
| `q`                | string  | Free text search over `title` and `subtitle`            |
| `price_max`        | int     | `price_from_qar <= price_max`                           |
| `age`              | string  | Listings whose `age_groups` JSON array contains the value (exact string match, e.g. `"6-10"`) |
| `lat` / `lng`      | float   | Enable distance annotation                              |
| `max_distance_km`  | float   | Filter by `distance_km <= value` (requires `lat`+`lng`) |
| `sort`             | enum    | `distance` (needs lat/lng), `rating`, `price_low`. Default: featured first, then newest. |
| `page`             | int     | Pagination                                              |

`ListingCard` response items (per `apps/catalog/serializers.py:6`):
```json
{
  "id": "7a3e...uuid",
  "title": "Lusail Arabic Tutoring",
  "category": "TUTORING",
  "subtitle": "1-on-1 sessions, KS1–KS3",
  "neighborhood": "Lusail",
  "lat": 25.43,
  "lng": 51.49,
  "rating": "4.8",
  "review_count": 12,
  "price_from_qar": 120,
  "image_urls": ["https://..."],
  "age_groups": ["6-10", "11-14"],
  "is_featured": true,
  "distance_km": 3.21
}
```

`distance_km` is `null` unless you sent `lat`+`lng`.

#### GET `/api/v1/listings/{id}/`

`ListingDetail` extends `ListingCard` with:
```json
{
  ...ListingCard fields,
  "description": "Long text...",
  "highlights": ["Native speaker", "Online or in person"],
  "owner_id": "42",
  "reviews": [
    {
      "id": "...uuid",
      "rating": 5,
      "text": "Amazing teacher.",
      "author_name": "Aisha",
      "created_at": "2026-06-10T15:00:00Z"
    }
  ]
}
```

The embedded `reviews` array is capped at the 10 newest. For the full paginated list, use `GET /api/v1/listings/{id}/reviews/` (§4.7).

**Private fields** (`session_schedule`, `exact_address`, `materials_required`) are **never** included on public catalog endpoints. They appear only via subscription detail (§4.5).

#### GET `/api/v1/categories/`

Returns counts of active listings per category, including zero-count categories (so the UI can render empty tabs):
```json
[
  {"key": "SCHOOLS", "count": 24},
  {"key": "NURSERIES", "count": 11},
  {"key": "ACTIVITIES", "count": 88},
  {"key": "ENTERTAINMENT", "count": 14},
  {"key": "TUTORING", "count": 47},
  {"key": "MASTERCLASSES", "count": 9},
  {"key": "PARTNERSHIPS", "count": 0}
]
```

### 4.3 Provider self-service (`apps/providers`)

All endpoints require auth + role `TUTOR` or `MASTERCLASS`.

| Method | Path                                       | Purpose                          |
| ------ | ------------------------------------------ | -------------------------------- |
| GET    | `/api/v1/provider/profile/`                | Own provider profile             |
| PATCH  | `/api/v1/provider/profile/`                | Update profile (PUT not allowed) |
| GET    | `/api/v1/provider/listings/`               | Own listings (all statuses)      |
| POST   | `/api/v1/provider/listings/`               | Create new listing               |
| GET    | `/api/v1/provider/listings/{id}/`          | Own listing detail               |
| PATCH  | `/api/v1/provider/listings/{id}/`          | Update own listing               |
| DELETE | `/api/v1/provider/listings/{id}/`          | Hard-delete own listing          |

#### Profile shape

```json
{
  "user_id": 42,
  "email": "tutor@example.com",
  "full_name": "Sara Ahmed",
  "role": "TUTOR",
  "is_verified": true,
  "display_name": "Sara — Arabic Tutor",
  "bio": "...",
  "subjects": ["Arabic", "English"],
  "hourly_rate_qar": 150,
  "availability": "Mon-Fri 4pm-8pm",
  "created_at": "...",
  "updated_at": "..."
}
```

Writable on PATCH: `display_name`, `bio`, `subjects` (string array), `hourly_rate_qar` (int, nullable), `availability`.

`is_verified` is **read-only** — admin flips it on the user, not here. On first GET the profile row is **lazy-created**, so the endpoint never 404s for a fresh provider.

#### Listing CRUD

Writable: `title`, `category`, `subtitle`, `neighborhood`, `lat`, `lng`, `price_from_qar`, `age_groups`, `image_urls`, `description`, `highlights`, `is_featured`, `session_schedule`, `exact_address`, `materials_required`.

Read-only: `id`, `owner_id`, `status`, `rating`, `review_count`, `created_at`, `updated_at`.

**Status is forced server-side** on every create/update:

| Provider `is_verified` | Resulting `status` |
| ---------------------- | ------------------ |
| true                   | `PENDING` (awaiting admin approval) |
| false                  | `DRAFT`  (provider can edit but can't publish) |

The admin moves listings `PENDING → ACTIVE` (or `REJECTED`) via Django admin. There is no provider-side "submit" or "publish" button — every successful save re-enters the workflow.

**Category constraint**: a TUTOR posting category `MASTERCLASSES` (or anything other than `TUTORING`) gets `400` with body `{"category": ["TUTOR providers can only create TUTORING listings."]}`.

### 4.4 Inquiries — the TUTORING booking flow (`apps/inquiries`)

Inquiries are the booking primitive for `TUTORING` listings. Both `FAMILY` (initiator) and `TUTOR` (recipient) sides exist.

#### State machine

```
NEW ─┬─► CONTACTED ─┬─► ACCEPTED ──► COMPLETED  (terminal)
     │              │
     └──────────────┴─► DECLINED                (terminal)
```

Transitions are performed by the provider only. Families create inquiries; providers move them through the funnel.

`contact_revealed` (boolean): see [§5 Billing gate](#5-billing-gate--phase-6) — currently always `true` in practice if `BILLING_GATE_ENABLED=false`, but the field is reserved.

#### Family endpoints

| Method | Path                                        | Purpose                       |
| ------ | ------------------------------------------- | ----------------------------- |
| GET    | `/api/v1/inquiries/`                        | Own inquiries (paginated)     |
| POST   | `/api/v1/inquiries/`                        | Create inquiry                |
| GET    | `/api/v1/inquiries/{id}/`                   | Detail                        |

POST body:
```json
{"listing_id": "uuid", "message": "Hi, my son needs help with Arabic..."}
```

Errors:
- 400 — listing is not `TUTORING` category, or non-`ACTIVE` status: `{"detail": "Listing is not available for inquiry"}`
- 404 — listing not found
- 403 — non-FAMILY role attempting to create

Response **201**: `FamilyInquiry` shape.

`FamilyInquiry` shape (no family contact, no provider contact):
```json
{
  "id": "uuid",
  "listing_id": "uuid",
  "provider_id": "42",
  "status": "NEW",
  "message": "...",
  "contact_revealed": false,
  "created_at": "...",
  "updated_at": "..."
}
```

#### Provider endpoints

| Method | Path                                                   | Effect                       |
| ------ | ------------------------------------------------------ | ---------------------------- |
| GET    | `/api/v1/provider/inquiries/`                          | Received inquiries           |
| GET    | `/api/v1/provider/inquiries/{id}/`                     | Detail                       |
| POST   | `/api/v1/provider/inquiries/{id}/contacted/`           | NEW → CONTACTED              |
| POST   | `/api/v1/provider/inquiries/{id}/accept/`              | NEW or CONTACTED → ACCEPTED  |
| POST   | `/api/v1/provider/inquiries/{id}/decline/`             | NEW or CONTACTED → DECLINED  |
| POST   | `/api/v1/provider/inquiries/{id}/complete/`            | ACCEPTED → COMPLETED         |

Transition actions are `POST` with no body. Response **200** with the updated `ProviderInquiry`.

List filter:
- `?status=NEW` (uppercase enum; lowercase tolerated)

`ProviderInquiry` shape (includes family block — but contact fields are gated):
```json
{
  "id": "uuid",
  "listing_id": "uuid",
  "provider_id": "42",
  "family": {
    "id": "17",
    "full_name": "Aisha Al-Thani",       // null while contact_revealed=false
    "phone": "+97455123456",             // null while contact_revealed=false
    "email": "aisha@example.com"         // null while contact_revealed=false
  },
  "status": "ACCEPTED",
  "message": "...",
  "contact_revealed": true,
  "created_at": "...",
  "updated_at": "..."
}
```

The `family` keys are **always present** with `null` values when contact is gated, so your client can render a stable schema.

Transition errors:
- 409 — invalid transition (e.g. `complete/` on a `NEW` inquiry): `{"detail": "Invalid transition: NEW → COMPLETED"}`
- 404 — inquiry doesn't exist or you aren't the provider

### 4.5 Subscriptions — the MASTERCLASSES auto-confirm flow (`apps/subscriptions`)

Subscriptions are the booking primitive for `MASTERCLASSES` listings. Auto-confirmed — no provider accept/decline.

#### State machine

```
CONFIRMED ──DELETE──► CANCELLED  (terminal — soft-delete, re-subscribe creates a new row)
```

#### Family endpoints

| Method | Path                                      | Effect                                    |
| ------ | ----------------------------------------- | ----------------------------------------- |
| GET    | `/api/v1/subscriptions/`                  | Own subscriptions                         |
| POST   | `/api/v1/subscriptions/`                  | Subscribe                                 |
| GET    | `/api/v1/subscriptions/{id}/`             | Detail (includes private listing fields)  |
| DELETE | `/api/v1/subscriptions/{id}/`             | Cancel (soft)                             |

POST body: `{"listing_id": "uuid"}`.

Errors:
- 400 — listing is not `MASTERCLASSES` category or non-`ACTIVE`
- 409 — already has a `CONFIRMED` subscription to this listing: `{"detail": "You already have an active subscription to this listing."}`
- 404 — listing not found

After cancellation the row is retained (status=`CANCELLED`, `cancelled_at` set). A subsequent POST creates a fresh row — the unique constraint only blocks duplicate `CONFIRMED` rows. So **re-subscribe is allowed** and is a brand-new resource.

`FamilySubscription` shape:
```json
{
  "id": "uuid",
  "listing_id": "uuid",
  "provider_id": "42",
  "status": "CONFIRMED",
  "cancelled_at": null,
  "created_at": "...",
  "updated_at": "...",
  "listing_private_details": {
    "session_schedule": "Saturdays 10am",
    "exact_address": "Building 7, Lusail Towers",
    "materials_required": ["notebook", "pen"]
  }
}
```

`listing_private_details` is the **value of subscribing** — the family gets logistics info that the public catalog does not expose.

#### Provider endpoints

| Method | Path                                              | Effect                                    |
| ------ | ------------------------------------------------- | ----------------------------------------- |
| GET    | `/api/v1/provider/subscriptions/`                 | Subscriber roster                         |
| GET    | `/api/v1/provider/subscriptions/{id}/`            | Detail                                    |

List filters: `?status=`, `?listing_id=`.

`ProviderSubscription` shape (no phone/email — see Phase 6):
```json
{
  "id": "uuid",
  "listing_id": "uuid",
  "listing_title": "Saturday Pottery for Kids",
  "family": {
    "id": "17",
    "full_name": "Aisha Al-Thani"
  },
  "status": "CONFIRMED",
  "cancelled_at": null,
  "created_at": "..."
}
```

### 4.6 Service suggestions (`apps/suggestions`)

Family-only "we'd love a pottery class in Lusail" requests. Admin manages the lifecycle in Django admin — there are **no admin REST endpoints**.

| Method | Path                                  | Effect                          |
| ------ | ------------------------------------- | ------------------------------- |
| GET    | `/api/v1/suggestions/`                | Own suggestions                 |
| POST   | `/api/v1/suggestions/`                | Submit a suggestion             |
| GET    | `/api/v1/suggestions/{id}/`           | Detail                          |

POST body:
```json
{
  "category": "ACTIVITIES",   // optional, defaults to ""
  "neighborhood": "Lusail",   // optional
  "message": "We'd love a pottery class for 7-year-olds."
}
```

Response (`SuggestionSerializer`):
```json
{
  "id": 12,
  "category": "ACTIVITIES",
  "neighborhood": "Lusail",
  "message": "...",
  "status": "NEW",
  "created_at": "...",
  "updated_at": "..."
}
```

`admin_notes` exists server-side but is **never returned** in any response. Don't try to read or write it.

Status values: `NEW`, `REVIEWED`, `ACTED_ON`, `DISMISSED`. Family-side is read-only after submission.

Only FAMILY may use these endpoints (`403` otherwise).

### 4.7 Reviews (`apps/reviews`)

| Method | Path                                                  | Auth          | Effect                                  |
| ------ | ----------------------------------------------------- | ------------- | --------------------------------------- |
| GET    | `/api/v1/listings/{listing_id}/reviews/`              | none          | Public list of reviews for a listing    |
| POST   | `/api/v1/listings/{listing_id}/reviews/`              | bearer+FAMILY | Create review (engagement-gated)        |
| GET    | `/api/v1/reviews/me/`                                 | bearer+FAMILY | Own reviews across all listings         |
| PATCH  | `/api/v1/reviews/{id}/`                               | bearer+FAMILY | Update own review (rating/text)         |
| DELETE | `/api/v1/reviews/{id}/`                               | bearer+FAMILY | Delete own review                       |

`Review` shape:
```json
{
  "id": "uuid",
  "rating": 5,
  "text": "Excellent",
  "author_name": "Aisha",     // falls back to "Anonymous" if full_name blank
  "created_at": "..."
}
```

#### Engagement gate (POST)

A family can only review a listing if there is evidence they engaged with it:

| Listing category | Requirement                                                   |
| ---------------- | ------------------------------------------------------------- |
| `TUTORING`       | Family has an inquiry in status `ACCEPTED` or `COMPLETED` for this listing |
| `MASTERCLASSES`  | Family has any subscription (active or cancelled) for this listing |
| Everything else  | No gate — any family may review                               |

Failing the gate returns **400** with:
```json
{"non_field_errors": ["You haven't engaged with this listing yet. ..."]}
```

Duplicate review (same listing + author) returns **409**:
```json
{"non_field_errors": ["You have already reviewed this listing."]}
```

#### Rating side-effect

A `post_save` / `post_delete` signal recomputes `Listing.rating` and `Listing.review_count` after every review change. Mobile doesn't need to do anything — just refetch the listing detail to see the new aggregate.

## 5. Billing gate — Phase 6

The mobile client should treat **billing as not built yet**. The current production behavior with `BILLING_GATE_ENABLED=False`:

- `Inquiry.contact_revealed` is auto-flipped to `true` when a provider accepts an inquiry (free-trial mode).
- The `ProviderInquiry.family` block returns real `full_name` / `phone` / `email` once `contact_revealed=true`.

When billing ships (see `docs/PHASE_6_BILLING.md`):
- `contact_revealed` will be gated on the family's subscription tier.
- The serializer shape will not change — only the values inside the `family` block.
- A new commission/billing surface will be added; details TBD.

Mobile should:
- Render the contact block defensively: if `contact_revealed=false`, show "contact details unlocked after acceptance" rather than crash on `null`.
- Not hardcode the assumption that contact is always returned on `ACCEPTED`.

## 6. Recommended client architecture

### 6.1 Token storage

- iOS: Keychain.
- Android: EncryptedSharedPreferences / Keystore-backed.
- Web: HttpOnly cookie is **not** an option here — the API expects `Authorization: Bearer`. Use IndexedDB or in-memory.

Persist `{token, expiry, user}` together. On launch, compare `expiry > now`. If expired or absent, send the user to login.

### 6.2 HTTP layer recommendations

- Set a 15-second connect timeout, 30-second read timeout.
- Inject `Authorization: Bearer {token}` via an interceptor.
- On `401` from any non-auth endpoint, drop the token and route to login. Don't retry.
- On `429`, surface `Retry-After` if present, otherwise back off 2s → 4s → 8s.

### 6.3 Pagination helper

Pages are full URLs in `next` / `previous`. Easiest pattern: just hit `next` verbatim (don't reconstruct).

### 6.4 Suggested screen → endpoint map

| Screen                          | Calls                                                                 |
| ------------------------------- | --------------------------------------------------------------------- |
| Splash / auth bootstrap         | `GET /auth/me/` (validates token)                                     |
| Sign up                         | `POST /auth/register/`                                                |
| Login                           | `POST /auth/login/`                                                   |
| Home — categories carousel      | `GET /categories/`                                                    |
| Category browse                 | `GET /listings/?category=TUTORING&lat=...&lng=...&sort=distance`      |

### 6.5 Push notifications

Push is **not yet** implemented on the backend. There's no device-registration endpoint and no outgoing push channel. Don't build a UI that promises push notifications. Polling on entry to a "my inquiries" screen is the current pattern.

## 7. Quirks and gotchas

1. **404 means "either missing or not yours."** Don't differentiate.
2. **Status enums are uppercase in responses, case-insensitive on input.** Always send uppercase to be safe.
3. **`is_featured` PATCH is server-controlled**: providers cannot self-promote. (Field is writable in the serializer but admin moderation overrides.)
4. **Provider listings always re-enter the workflow on every save.** A provider editing an `ACTIVE` listing flips it back to `PENDING`. Warn the user in the UI before submitting an edit.
5. **Cancel + re-subscribe creates a new subscription row**, with a new `id`. Don't try to "reactivate" an old one.
6. **Categories are an enum; never let mobile invent new ones.** The list is: `SCHOOLS`, `NURSERIES`, `ACTIVITIES`, `ENTERTAINMENT`, `TUTORING`, `MASTERCLASSES`, `PARTNERSHIPS`.
7. **No file upload endpoint.** `image_urls` must be pre-uploaded by the client to your chosen storage. Coordinate with backend before building an upload flow.
8. **No real-time channel.** No websockets, no SSE. Use polling for inquiry/subscription updates.
9. **Password changes do not invalidate tokens.** If you need that UX (e.g. "log out other devices on password change"), call `/auth/logoutall/` explicitly after the PATCH.
10. **`rating` is a decimal string, not a number.** Parse with care — `"4.0"`, not `4.0`.

## 8. Versioning & deprecation policy

- The API path includes `/v1/`. Breaking changes will go to `/v2/` and be announced in this doc.
- Additive changes (new optional fields, new endpoints) may land on `/v1/` without notice. The client should ignore unknown JSON keys gracefully.
- Server-driven URLs (`next`, `previous` in pagination) are stable and should be followed verbatim — don't reconstruct them.

## 9. Quick start checklist for the mobile agent

1. Pull the OpenAPI spec: `docker compose exec backend python manage.py spectacular --file schema.yml`.
2. Generate a typed client from `schema.yml` (e.g. `openapi-generator-cli`, `swift-openapi-generator`, or `openapi-typescript-codegen`).
3. Implement an auth interceptor that:
   - Adds `Authorization: Bearer {token}` to every request when a token is stored.
   - Listens for `401` and routes to login.
4. Build the registration / login screens against §4.1.
5. Build the public catalog screens against §4.2.
6. Branch on `user.role`:
   - `FAMILY` → inquiry / subscription / review / suggestion flows.
   - `TUTOR` / `MASTERCLASS` → provider profile + listing CRUD + received inquiries/subscriptions.
7. Always treat enums as strings, IDs as opaque strings, timestamps as ISO-8601 UTC.
